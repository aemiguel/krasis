11. Local src/gpu_decode.rs restore of graphed legacy replay path
Classification: NEGATIVE (was incorrectly marked IMPROVING)
The local restore has THREE specific bugs vs the b8c071d baseline:

  BUG A: Weights written to wrong buffer
  - Old code: packs weights into h_batch_upload at ptr_stride*4 offset, uploads everything
    to d_batch_upload with ONE cuMemcpyHtoDAsync. The captured graph's multi_expert_weighted_add
    kernel reads weights from d_batch_upload + ptr_stride*4.
  - Current code: writes weights to d_batch_weights (a SEPARATE allocation). The graph's GEMV
    kernels still read from d_batch_upload + ptr_stride*4, so they read STALE/ZERO weights.
  - Impact: CORRECTNESS BUG. MoE accumulation uses wrong weights every token. This alone
    could explain degraded output quality and potentially affect routing/EOS behavior.

  BUG B: 5 separate H2D copies instead of 1 contiguous copy
  - Old code: packs all 4 pointer arrays + weights into h_batch_upload (contiguous), then
    does ONE cuMemcpyHtoDAsync of upload_bytes to d_batch_upload.
  - Current code: does 5 separate cuMemcpyHtoDAsync calls from 5 different host arrays.
  - Impact: NEGATIVE for performance. 5x more driver API calls per MoE layer. With 48 MoE
    layers, that's ~240 extra driver API calls per token. Small H2D transfers have high
    per-call overhead in the CUDA driver.

  BUG C: Dummy expert uses single base pointer for all 4 slot types
  - Old code: reads 4 properly-offset dummy pointers from d_dummy_ptrs (w13p, w13s, w2p, w2s
    each point to correct sub-regions of the dummy buffer).
  - Current code: uses raw d_dummy_expert base for ALL 4 slots. Minor, but the captured graph's
    GEMV kernels expect properly-separated sub-buffer offsets.
  - Impact: Low risk of crash (reads from valid memory), but could produce wrong intermediate
    values for the dummy expert slots.
